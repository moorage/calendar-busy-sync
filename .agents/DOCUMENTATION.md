  - `docs/ideas/backlog/booking-appointment-management-ux-ia-audit.md` records the June 3, 2026 audit of appointment-type CRUD, booking setup, publishing, verification, Vercel deployment, page customization, n=0/n=5 states, progressive disclosure, and iconography. The audit was completed through `docs/exec-plans/completed/2026-06-03-booking-ux-ia-refresh.md`, which collapsed the separate setup sheet into one Booking workspace, added explicit local/generated/uploaded/live/verified evidence, added appointment-type pause/resume, surfaced publish/deploy evidence, and made page customization and Vercel deployment guidance first-class native tasks.
  - the Booking settings entry opens one workspace overview instead of a parallel setup sheet; appointment types have lifecycle badges plus pause/resume; duplicated types start paused; paused types stay out of public config and slots; page generation/upload/verification records generated, uploaded, served, and verified evidence; `Live` requires a served public-config fingerprint match; and the public page handles n=0, n=1, n=5, n=20, no-slots, local-preview, compact timezone disclosure, and 15-minute slot rounding.
  - the Public Page workspace now persists native public name, title, subtitle, timezone, and theme colors; those fields feed generated public artifacts and the generated fingerprint. It also exposes safe customization files, protected protocol files, and a fresh-generating preview action.
  - the Request Inbox workspace now supports app-managed Vercel setup. Vercel setup asks for a Vercel token, project ID/name, and optional team ID/slug, then generates the inbox admin token, deploys the bundled relay template through Vercel REST, saves the returned inbox URL, compares `/healthz` `allowedOrigin` evidence to the public page origin, and distinguishes configured, reachable, origin-mismatch, and ready states without logging secrets.
  - the Page and Publish workspaces now show last-action feedback in-place, open the generated local preview through AppKit on macOS, and emphasize the current lifecycle action: generate when no local files exist, publish after local generation, and verify after upload while GitHub Pages is catching up.
  - appointment types now use a list-to-detail editing flow with Back navigation. Each appointment type owns its accepted-booking calendar target; approval/decline writes and Google Meet gating use that per-type target, while target calendar IDs stay out of public artifacts and public fingerprints.
  - public page templates now have an end-user path: the app bundles the seed `booking-site` folder, seeds a persistent `BookingSiteTemplate` folder in Application Support, exposes `Open template folder`, and generates publishable page files from that editable folder instead of from a developer checkout path.
  - GitHub Pages publishing is now root-only for a dedicated empty repository: the Publish workspace no longer asks for a remote folder, the publisher uploads generated files at repository root, and preflight blocks roots that contain non-generated files.
  - GitHub Pages publishing now uses deploy keys only: the Publish workspace generates a repository-specific Ed25519 keypair, stores the private key in secure storage, shows/copies the public key for GitHub deploy-key setup with write access, verifies SSH reachability, and publishes generated root files through Git over SSH.
  - iCloud shared settings now sync non-secret native Booking setup: appointment types, page/inbox URLs, repository/branch, Vercel metadata, profile/theme fields, selected appointment type, and automatic-approval preferences. Vercel account tokens, booking private/signing keys, inbox admin tokens, deploy-key private keys, generated page-file paths, and editable HTML/CSS/template files stay local.
  - each appointment type now owns its public availability horizon; the native editor exposes `Show availability`, the model caps it at three months, and manual/background poll publishing regenerates every active appointment type across its configured horizon.
  - macOS GitHub Pages publishing no longer treats `/usr/bin/git`, `/usr/bin/ssh`, or `/usr/bin/ssh-keygen` as production defaults. The app resolves bundled `booking-git`, `booking-ssh`, and `booking-ssh-keygen` helpers for the deploy-key path in local helper-enabled builds and reports a build/package error when those helpers are absent.
  - local macOS app builds can package the Git/SSH helper toolchain through `scripts/copy-booking-git-toolchain`, but App Store archive/install builds now remove those copied Apple helper binaries after App Review flagged build `1.4 (7)` for non-public/deprecated API references. The reviewed app uses the existing missing-helper error path until GitHub publishing moves to a review-safe implementation.
  - app-managed Vercel inbox deploy now provisions the required Blob storage path: the app resolves the Vercel project, reuses a connected Blob store when present, creates and connects a public Blob store to production when absent, deploys the relay template, and then verifies `/healthz`.
  - Booking configuration failures now bridge through `LocalizedError`, so GitHub deploy-key verification and Pages publish failures show actionable audit messages instead of raw Swift enum error codes.
  - Missing local GitHub deploy-key private keys and missing bundled Git/SSH helpers are now shown directly in Booking action messages; this covers shared setup metadata arriving on a Mac whose Keychain does not contain the local private key.
